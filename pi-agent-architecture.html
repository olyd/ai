pi agent 源码与架构导读 v0.75.4

仓库：earendil-works/pi-mono ｜ 作者：Mario Zechner

这份导读基于真实源码逐文件走读，目标是让读者在两小时内建立可以独立扩展 pi 的心智模型。

0. 这是什么项目

pi 是一个可自扩展的交互式编程 agent CLI，定位类似 Claude Code / Codex / aider。它把"agent 是什么"拆成了非常干净的三层：

  • @earendil-works/pi-ai — 多 provider LLM 适配层（OpenAI/Anthropic/Google/Bedrock…），统一成一个流式事件协议。
  • @earendil-works/pi-agent-core — 与 LLM 解耦的 agent 主循环：调模型、调工具、回灌结果、会话持久化、上下文压缩。
  • @earendil-works/pi-coding-agent — 在 agent-core 之上塞进 7 个真实的编辑器工具（read / bash / edit / write / grep / find / ls），加上 TUI、扩展系统、斜杠命令。

外加 @earendil-works/pi-tui 提供 differential rendering 的终端 UI 原语。整个 mono repo 用 Node ≥22、原生 TypeScript（Node strip-only mode）、TypeBox 做参数 schema。

读这份代码的最大启发是："agent loop"本身其实很短（一个 while-while 双循环 + 一个流转换器），真正复杂的是会话状态、压缩、并发安全和扩展点。这也是 pi 的代码组织方式所强调的。

1. monorepo 总览

包依赖关系

flowchart LR TUI[pi-tui
终端原语] AI[pi-ai
provider 抽象 + 流] CORE[pi-agent-core
agent loop + session + compaction] CA[pi-coding-agent
CLI + 工具 + 扩展] AI --> CORE CORE --> CA TUI --> CA AI --> CA

构建顺序在根 package.json 中写死：tui → ai → agent → coding-agent。这种线性依赖让每一层都能单独 unit test，也让 SDK 用户可以只依赖 pi-agent-core 嵌入到自己的应用里（如 Slack bot），不必拖进 TUI。

启动链路

pi 命令的入口是 packages/coding-agent/dist/cli.js（源在 src/cli.ts）。一条用户输入抵达 LLM 的路径如下：

sequenceDiagram participant U as 用户 participant CLI as cli.ts
(args, mode) participant RT as AgentSessionRuntime participant AS as AgentSession participant H as AgentHarness participant L as agentLoop participant AI as pi-ai.streamSimple U->>CLI: pi "改下这个 bug" CLI->>RT: createAgentSessionRuntime(cwd, agentDir) RT->>AS: new AgentSession({ tools, model, session }) AS->>H: new AgentHarness(...) AgentSessionRuntime（src/core/agent-session-runtime.ts）的作用是 "持有当前 session 及其 cwd-bound services，并支持原地替换"—— 当用户 /new、/resume 或更换工作目录时，runtime 会先 teardown 旧 services 再创建新的，避免上一会话的 file mutation queue、扩展钩子泄漏到下一会话。

2. pi-ai：LLM 抽象层

pi-ai 的目标：把 10 多家 provider 的非标准协议封成一套 AsyncIterable 事件流，外层只需对 11 种事件做模式匹配。packages/ai/src/providers/ 下每个 provider 是一个独立文件，全部走 streamSimple 这个统一入口。

注意三件事：

  • 每个 delta 事件都携带 partial—— 完整快照。UI 直接渲染 partial 就是当前消息状态，不必自己 reduce。
  • thinking、toolcall 和 text 是平行内容块，靠 contentIndex 区分；这样能同时支持 Claude 的 thinking 块、OpenAI 的 reasoning 字段、并行 tool calls。
  • stream 永远以 done 或 error 结尾—— 这是下面那个"不抛异常"契约的基础。

这条契约非常关键：provider 失败必须通过事件流报告，而不是 throw。这样 agent loop 不用区分"请求失败"和"模型说我做完了"两种代码路径，只需要一个 for-await。整个错误恢复、UI 渲染、abort 处理都因此变得线性可读。

3. pi-agent-core：Agent 主循环

这一节的全部肉都在 packages/agent/src/agent-loop.ts（~740 行）。它做四件事：

  1. 把 AgentMessage[]（含自定义消息）压平成 LLM 能吃的 Message[]；
  2. 调 provider，把流事件转成 AgentEvent 转发；
  3. 从 assistant 消息里抠出 toolCalls，调用工具，构造 ToolResultMessage 回灌；
  4. 处理 steering / followUp 队列，让用户能"中途插话"或"agent 停了再追问"。

while (true) {                       // 外循环：等 follow-up
  let hasMoreToolCalls = true;
  while (hasMoreToolCalls || pendingMessages.length > 0) {  // 内循环：一个会话回合
    if (!firstTurn) await emit({ type: "turn_start" });
    // 1. 注入 steering message（用户在 agent 跑时打字）
    for (const m of pendingMessages) currentContext.messages.push(m);
    // 2. 调 LLM，等流结束拿到完整 AssistantMessage
    const message = await streamAssistantResponse(currentContext, config, signal, emit, streamFn);
    if (message.stopReason === "error" || message.stopReason === "aborted") {
      await emit({ type: "agent_end", messages: newMessages }); return;
    }
    // 3. 取 toolCalls 执行（顺序或并行）
    const toolCalls = message.content.filter(c => c.type === "toolCall");
    if (toolCalls.length) {
      const batch = await executeToolCalls(currentContext, message, config, signal, emit);
      hasMoreToolCalls = !batch.terminate;
      for (const r of batch.messages) currentContext.messages.push(r);
    } else hasMoreToolCalls = false;
    // 4. 给宿主一个换模型 / 重建 context 的机会
    const snap = await config.prepareNextTurn?.({ message, ... });
    if (snap) { currentContext = snap.context ?? currentContext; ... }
    if (await config.shouldStopAfterTurn?.({...})) { await emit({ type: "agent_end", ... }); return; }
    pendingMessages = (await config.getSteeringMessages?.()) || [];
  }
  // 内循环结束：看用户有没有"等 agent 停了再说"的消息
  const followUps = (await config.getFollowUpMessages?.()) || [];
  if (followUps.length) { pendingMessages = followUps; continue; }
  break;
}

流式 → 完整消息

// packages/agent/src/agent-loop.ts:275
async function streamAssistantResponse(context, config, signal, emit, streamFn) {
  let messages = context.messages;
  if (config.transformContext) messages = await config.transformContext(messages, signal);
  const llmMessages = await config.convertToLlm(messages);     // AgentMessage → LLM Message
  const llmContext = { systemPrompt: context.systemPrompt, messages: llmMessages, tools: context.tools };
  const response = await (streamFn || streamSimple)(config.model, llmContext, { ...config, signal });

  let partialMessage: AssistantMessage | null = null;
  for await (const event of response) {
    switch (event.type) {
      case "start":
        partialMessage = event.partial;
        context.messages.push(partialMessage); // 先把"骨架"塞进 context
        await emit({ type: "message_start", message: { ...partialMessage } });
        break;
      case "text_delta": case "thinking_delta": case "toolcall_delta":
      case "text_start": case "text_end": /* ...等等 */
        partialMessage = event.partial;
        context.messages[context.messages.length - 1] = partialMessage; // 原地替换
        await emit({ type: "message_update", assistantMessageEvent: event, message: { ...partialMessage } });
        break;
      case "done": case "error": {
        const finalMessage = await response.result();
        context.messages[context.messages.length - 1] = finalMessage;
        await emit({ type: "message_end", message: finalMessage });
        return finalMessage;
      }
    }
  }
}

工具执行：序列 / 并行 / 早停

// packages/agent/src/agent-loop.ts:373
async function executeToolCalls(currentContext, assistantMessage, config, signal, emit) {
  const toolCalls = assistantMessage.content.filter(c => c.type === "toolCall");
  // 任何一个工具声明自己是 sequential，整批就退化成 sequential
  const hasSeq = toolCalls.some(tc =>
    currentContext.tools?.find(t => t.name === tc.name)?.executionMode === "sequential");
  if (config.toolExecution === "sequential" || hasSeq) {
    return executeToolCallsSequential(currentContext, assistantMessage, toolCalls, config, signal, emit);
  }
  return executeToolCallsParallel(currentContext, assistantMessage, toolCalls, config, signal, emit);
}

function shouldTerminateToolBatch(finalized) {
  // 只有"全部"工具都说 terminate:true 才早停，避免一个工具把整轮拉死
  return finalized.length > 0 && finalized.every(f => f.result.terminate === true);
}
packages/agent/src/agent-loop.ts:373-388, 544-546

并行执行的实现也很干净：先顺序地做 prepareToolCall（参数验证、beforeToolCall 钩子），把每个准备好的 call 包成 lambda，最后 Promise.all 一起跑。tool-result 消息的顺序仍然按 assistant 源顺序追加 —— 这一点很重要，否则 LLM 在下一轮会被乱序的 toolResult 弄崩。

Before/After 钩子（沙箱化的入口）

// packages/agent/src/agent-loop.ts:562
async function prepareToolCall(currentContext, assistantMessage, toolCall, config, signal) {
  const tool = currentContext.tools?.find(t => t.name === toolCall.name);
  if (!tool) return { kind: "immediate", result: createErrorToolResult(`Tool ${toolCall.name} not found`), isError: true };
  try {
    const prepared = prepareToolCallArguments(tool, toolCall);
    const validatedArgs = validateToolArguments(tool, prepared);   // TypeBox schema validation
    if (config.beforeToolCall) {
      const before = await config.beforeToolCall({ assistantMessage, toolCall, args: validatedArgs, context: currentContext }, signal);
      if (before?.block) return { kind: "immediate", result: createErrorToolResult(before.reason || "..."), isError: true };
    }
    return { kind: "prepared", toolCall, tool, args: validatedArgs };
  } catch (error) {
    return { kind: "immediate", result: createErrorToolResult(String(error)), isError: true };
  }
}
packages/agent/src/agent-loop.ts:562-626

beforeToolCall 返回 { block: true, reason } 就能阻止任何工具执行；afterToolCall 可以重写 content / details / isError / terminate。这两个钩子加上 transformContext 是 sandbox 扩展和审计扩展的全部支撑面—— 比如 Anthropic sandbox extension（在 examples/extensions/sandbox 里）就是用 beforeToolCall 把 bash 调用劫持进 firejail。

AgentHarness：业务编排器

agent-loop.ts 是无状态函数。真正"长期持有 session、抗中断、能改模型、能 abort"的是 AgentHarness（packages/agent/src/harness/agent-harness.ts，~1000 行）：

几个值得抄走的设计：

  • createStreamFn 闭包绑定 turnState。每次 LLM 调用之前，harness 都重新解析一次 getApiKeyAndHeaders(model)，专门为 GitHub Copilot 这种"OAuth token 在长 tool 执行期间会过期"的场景做的。

  • emitHook 是有返回值的事件：扩展可以在 context hook 里替换整段消息历史、在 tool_call hook 里 block、在 session_before_compact 里取消或者自己实现 compaction —— 全部统一在 handlers Map 上。

4. 会话：树状结构与压缩

为何是树而非线性

很多 agent 把会话存成 messages: AgentMessage[]。pi 不是 —— 它把每条事件叫做一个 SessionTreeEntry，每个 entry 都带 parentId，构成一棵树：

graph TD root[session_info] m1[user: 帮我修 bug] a1[assistant: 看了一下，应该改 foo.ts] t1[toolResult: edit foo.ts ok] a2[assistant: 改好了，跑测试] m2[user 重写: 不对，先看 bar.ts] a3[assistant: 看了 bar.ts] cs[compaction summary] m3[user: 接着上面的来] root --> m1 --> a1 --> t1 --> a2 a1 --> m2 --> a3 a3 --> cs --> m3 classDef sum fill:#fff8c5,stroke:#9a6700 class cs sum

类型一共九种（SessionTreeEntry）：message / custom_message / thinking_level_change / model_change / compaction / branch_summary / label / session_info / tool_use。每个 entry 由 jsonl-storage 追加写入磁盘（一行一个 entry），所以会话文件天然支持 tail -f 调试。

"当前位置"由 leafId 标记。/fork 从某条 user 消息分出新枝、/clone 复制当前位置、/tree 在分支间跳转，这些操作只需要改 leafId，原有 entry 一行都不动 —— git 式的状态管理。

buildSessionContext

// packages/agent/src/harness/session/session.ts:21
export function buildSessionContext(pathEntries: SessionTreeEntry[]): SessionContext {
  let thinkingLevel = "off";
  let model: { provider: string; modelId: string } | null = null;
  let compaction: CompactionEntry | null = null;
  // 第一遍：从 root → leaf 扫一遍，记下最后一次模型/思考切换、最后一个 compaction 边界
  for (const entry of pathEntries) {
    if (entry.type === "thinking_level_change") thinkingLevel = entry.thinkingLevel;
    else if (entry.type === "model_change")     model = { provider: entry.provider, modelId: entry.modelId };
    else if (entry.type === "message" && entry.message.role === "assistant")
      model = { provider: entry.message.provider, modelId: entry.message.model };
    else if (entry.type === "compaction")       compaction = entry;
  }
  // 第二遍：从最后一个 compaction 边界之后挑出真正要喂给 LLM 的消息
  if (compaction) {
    messages.push(createCompactionSummaryMessage(compaction.summary, compaction.tokensBefore, compaction.timestamp));
    /* 把 firstKeptEntryId 之后的 entries 加进去 */
  }
  return { messages, thinkingLevel, model };
}

这是 pi 把"会话树"翻译回"线性 LLM context"的关键。compaction 不会真删掉历史 entries，只是写一个 compaction entry 把它们"挡在外面"，浏览树时仍能完整看到原貌。这是相当多 agent 系统会踩的坑（删了就找不回来），pi 用 append-only + 边界标记解决了。

上下文 compaction

默认设置：reserveTokens: 16k（留给 summary prompt 和输出）、keepRecentTokens: 20k（保留最近多少 token）。逻辑在 packages/agent/src/harness/compaction/compaction.ts，三步：

  1. prepareCompaction：在 branch 上从后往前数 token，找到要"砍掉"的分界点 firstKeptEntryId。
  2. compact：把砍掉那段对话序列化（含工具调用细节），用同模型 + custom instructions 跑一次 completeSimple，要求输出一个 summary。
  3. 追加一个 CompactionEntry（含 summary 文本、firstKeptEntryId、tokensBefore、读过/改过的文件列表）。

用户在 /compact 时如果给了 customInstructions，AgentHarness 还会先触发 session_before_compact hook，扩展甚至能整段接管自己实现压缩。文件列表 { readFiles, modifiedFiles } 之所以单独存，是为了让下次 compaction 链式继承——避免反复压缩后丢失"这个 session 改过哪些文件"的元信息。

// packages/coding-agent/src/core/tools/file-mutation-queue.ts
const fileMutationQueues = new Map<string, Promise<void>>();

function getMutationQueueKey(filePath: string): string {
  const resolvedPath = resolve(filePath);
  try { return realpathSync.native(resolvedPath); } catch { return resolvedPath; }
}

export async function withFileMutationQueue<T>(filePath: string, fn: () => Promise<T>): Promise<T> {
  const key = getMutationQueueKey(filePath);
  const currentQueue = fileMutationQueues.get(key) ?? Promise.resolve();
  let releaseNext!: () => void;
  const nextQueue = new Promise<void>((r) => { releaseNext = r; });
  const chainedQueue = currentQueue.then(() => nextQueue);
  fileMutationQueues.set(key, chainedQueue);
  await currentQueue;
  try { return await fn(); }
  finally {
    releaseNext();
    if (fileMutationQueues.get(key) === chainedQueue) fileMutationQueues.delete(key);
  }
}

为什么要写这个？—— 因为 agent loop 默认 parallel 执行工具。如果模型在一轮里发了两个 edit 都改 foo.ts，并行的 read-modify-write 会丢更新。withFileMutationQueue 用 realpath（解析符号链接）作为 key 把同一物理文件的 mutation 排成单链，不同文件还能并行 —— 是非常实用的并发原语。

bash 工具：streaming + abort + 进程树清理

// packages/coding-agent/src/core/tools/bash.ts:65 (节选)
export function createLocalBashOperations(options?: { shellPath?: string }): BashOperations {
  return {
    exec: (command, cwd, { onData, signal, timeout, env }) => new Promise((resolve, reject) => {
      const { shell, args } = getShellConfig(options?.shellPath);
      const child = spawn(shell, [...args, command], {
        cwd, detached: process.platform !== "win32",
        env: env ?? getShellEnv(),
        stdio: ["ignore", "pipe", "pipe"], windowsHide: true,
      });
      if (child.pid) trackDetachedChildPid(child.pid);
      let timedOut = false;
      const timeoutHandle = timeout ? setTimeout(() => {
        timedOut = true; if (child.pid) killProcessTree(child.pid);
      }, timeout * 1000) : undefined;
      child.stdout?.on("data", onData);
      child.stderr?.on("data", onData);
      const onAbort = () => { if (child.pid) killProcessTree(child.pid); };
      if (signal) signal.aborted ? onAbort() : signal.addEventListener("abort", onAbort, { once: true });
      // waitForChildProcess 是 pi 自己写的：避免 detached 子进程持有的 stdio 让父进程 hang
      waitForChildProcess(child).then((code) => { /* 清理 + resolve */ });
    }),
  };
}
packages/coding-agent/src/core/tools/bash.ts:65-130

三处细节：

  • detached: true + killProcessTree(pid) —— 不只杀 shell，还杀 shell fork 出来的 npm/python/...，否则 Esc 中断后还在跑。
  • onData 流式回调，让 UI 能边跑边显示输出。OutputAccumulator（同目录）负责按 DEFAULT_MAX_BYTES / DEFAULT_MAX_LINES 截断 + 落盘 fullOutputPath，避免 LLM 上下文被 npm install 撑爆。
  • 把 BashOperations 抽出来做接口 —— SSH 扩展、Docker 扩展、远端沙箱都能替换它而不动 bash 工具本身。

  • 注册 provider：registerProvider(...) —— 第三方厂商支持的接入点
  • hook 事件：before_agent_start / context / tool_call / tool_result / session_before_compact / session_before_tree / before_provider_request / before_provider_payload / after_provider_response
  • UI：调用 tui.overlay() 弹窗、注册 widget、自定义渲染器

8. 整体总结与设计亮点

flowchart TB subgraph TUI[pi-tui] T[differential renderer] end subgraph AI[pi-ai] SS[streamSimple] PR[providers/*] EV[AssistantMessageEventStream] SS --> EV PR --> SS end subgraph CORE[pi-agent-core] AL[agent-loop.ts] H[AgentHarness] SES[Session

4 个关键抽象

  1. AssistantMessageEventStream：用一个 11 种事件的统一协议把所有 provider 拍平。失败也是事件，所以上层一个 for-await 全搞定。
  2. "convertToLlm" 回调：让 host 自定义消息类型，agent-core 只在 LLM 调用边界做一次翻译，UI 和 session 可以承载任意伪消息。
  3. Session 是 append-only 树：分支、克隆、压缩都不破坏历史；buildSessionContext 把树投影回线性 context。
  4. Definition / Tool 双层 + Operations 接口：工具元数据归 definition，实际执行通过 BashOperations/EditOperations 注入，给 SSH/sandbox/Docker 留口子。

区别于其他 coding agent 的设计选择

  • 三队列 steering / followUp / nextTurn：把"用户什么时候能插话"这件事建模成了 3 种语义不同的队列，比单一阻塞输入更像 IDE。
  • 失败不抛异常的 stream 契约：少见但优雅，让错误处理代码量大幅缩减。
  • file mutation queue 用 realpath 做 key：解决并行 edit 的 race condition，且能跨符号链接正确合并。
  • compaction 是 entry 而非删除：保留可回溯性，并把"已读/已改文件列表"链式传递。
  • 扩展通过 jiti + virtualModules 加载：让用户脚本能直接 import { Type } from "typebox" 而不需要任何 build，同时在编译后单二进制里也成立。
  • 无 agent 子任务 / 子 agent 概念：pi 只有一个 agent loop。需要"分支"时用 session tree + branch summary 代替，比 sub-agent 简单很多。
